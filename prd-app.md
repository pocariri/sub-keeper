# PRD (확장본) — 구독 관리 서비스 · 웹 + 모바일 앱

> **문서 목적**: 실제 제품(구독 관리 앱)의 제품·기술 요구사항 정의.
> Claude Code가 이 문서를 기준으로 모노레포를 스캐폴딩하고 화면을 구현할 수 있도록 작성한다.
> 마케팅 **소개(랜딩) 사이트**는 별도 문서(`prd.md`)에서 다룬다.

---

## 1. 제품 개요

| 항목 | 내용 |
|------|------|
| 제품 | 여러 구독을 한곳에서 관리하는 서비스 (가칭 `[제품명]`) |
| 핵심 가치 | 흩어진 구독의 결제일·총액을 자동 정리하고, 갱신 전에 알려준다 |
| 타겟 | OTT·클라우드·멤버십 등 여러 구독을 쓰는 일반 사용자 |
| 플랫폼 | 웹 + 모바일 앱(iOS·Android) 동시 지원 |
| 백엔드 | BaaS 한 벌을 웹·모바일이 공유 |

## 2. 플랫폼 & 아키텍처

### 2.1 원칙
- **백엔드는 한 벌, UI는 플랫폼별로 따로, 로직·타입은 공유.**
- 웹과 모바일은 각자 목적에 맞는 UI를 갖는다. (모바일은 데스크탑 축소가 아니라 별도 레이아웃)
- 반복 로직(결제 주기 계산, 다음 결제일 산정, 통화 환산, 카테고리 집계, 유효성 검증)은 공유 패키지에 한 번만 구현한다.

### 2.2 기술 스택

| 레이어 | 선택 | 비고 |
|--------|------|------|
| 웹 | React + Vite | 로그인 뒤 앱 화면(SPA) |
| 모바일 | Expo (React Native) | iOS·Android 단일 코드, EAS로 스토어 배포 |
| 공유 | TypeScript 패키지 | 타입·비즈니스 로직·BaaS 클라이언트·zod 검증 |
| 백엔드 | BaaS (Supabase 권장) | 인증·DB·스토리지·함수. 대안: Firebase |
| 모노레포 도구 | pnpm workspaces 또는 Turborepo | 빌드·의존성 관리 |
| 차트 | 웹: recharts / 모바일: victory-native 등 | 로직은 공유, 렌더링은 플랫폼별 |

### 2.3 모노레포 폴더 구조 (초안)

```
repo/
├─ apps/
│  ├─ web/                # React + Vite (앱 화면 + 관리자 페이지)
│  └─ mobile/             # Expo (React Native)
├─ packages/
│  └─ core/               # 공유 패키지
│     ├─ types/           # Subscription, User, Plan 등 타입
│     ├─ logic/           # 결제일 계산, 통화 환산, 카테고리 집계
│     ├─ validation/      # zod 스키마
│     └─ baas/            # BaaS 클라이언트 래퍼 (auth, db, push)
├─ package.json
└─ turbo.json (또는 pnpm-workspace.yaml)
```

> 공유하는 것: **로직·타입·BaaS 접근**. 공유하지 않는 것: **UI 컴포넌트**(웹=DOM/CSS, 모바일=RN 컴포넌트).

## 3. 백엔드 (BaaS)

### 3.1 선택
- **권장: Supabase(Postgres)** — 카테고리별/기간별 집계, 연간 리포트 등 관계형 쿼리가 편함. (⚠️ 확인 필요)
- **대안: Firebase** — 데이터가 단순하고 푸시(FCM) 통합을 우선한다면 적합.
- 두 옵션 모두 소셜+이메일 로그인, 웹·React Native SDK 지원.

### 3.2 데이터베이스 스키마 초안

```
profiles
  id (uuid, = auth user id, PK)
  email
  plan            enum('free','premium')  default 'free'
  premium_until   timestamp (nullable)
  created_at

subscriptions
  id (uuid, PK)
  user_id (uuid, FK → profiles.id)
  service_name    text
  amount          numeric
  currency        text          # 'KRW','USD' ...
  billing_cycle   enum('weekly','monthly','yearly','custom')
  custom_days     int (nullable)
  next_billing_at date
  category        text          # 아래 3.4 기본 목록
  payment_method  text
  memo            text (nullable)
  is_active       bool default true
  created_at / updated_at

exchange_rates            # 선택: 결제일 기준 환율 캐시
  base_currency
  quote_currency
  rate
  as_of_date

service_presets           # 전 사용자 공유 참조 데이터(읽기 전용) — 구독 추가 시 입력 미리 채우기
  id (uuid, PK)
  name                  text unique   # 서비스명(검색/자동완성 키): 'Netflix' ...
  category              text          # 3.4 카테고리 라벨 문자열 → 앱에서 CategoryId 로 매핑
  default_amount        numeric (nullable)  # 대표 요금(예시값, 사용자가 추가 시 수정 가능)
  default_currency      text default 'KRW'
  default_billing_cycle enum('weekly','monthly','yearly','custom') default 'monthly'
  logo_url              text (nullable)     # 로고(선택, 추후)
  is_active             bool default true
  sort_order            int default 100     # 작을수록 목록 상단
  created_at
```

- **접근 제어(RLS 등)**: 각 사용자는 본인 `subscriptions`만 읽고 쓸 수 있어야 한다.
- **`service_presets`**: 로그인 사용자 전체가 **활성 항목을 읽기만** 가능(쓰기 정책 없음 → 클라이언트에서 수정 불가). 시드/갱신은 서비스 롤(관리자)만. `default_amount` 는 예시값이며 실제 요금은 수시로 바뀔 수 있어 사용자가 추가 시 직접 수정한다. 마이그레이션: `supabase/migrations/0002_service_presets.sql`.
  - 조회는 `packages/core` 의 BaaS 클라이언트 경유(`listServicePresets`), 구독 추가 흐름은 4.1 및 `prd-landing.md 4.3` 참조.
- **관리자 지표**는 `profiles` / `subscriptions` / 결제 기록을 집계해서 산출 (9번 참조).

### 3.3 인증
- **소셜 로그인 + 아이디/비밀번호 로그인 둘 다 지원.**
- 모바일은 소셜 로그인 시 리다이렉트/딥링크 설정이 웹과 다름 → BaaS SDK가 대부분 처리하지만 초기 세팅에서 확인 필요.

### 3.4 기본 카테고리 (드롭다운)
엔터테인먼트(OTT·음악), 생산성/업무, 클라우드/저장소, 교육, 뉴스/미디어, 게임, 쇼핑/멤버십, 건강/피트니스, 금융, 기타

### 3.5 통화 / 환율
- 통화가 섞여도 총액을 합산해 보여준다.
- 환산 기준: **결제일 기준 환율.**
- 환율 데이터 소스 필요 (외부 API 또는 수동 입력) → 오픈 이슈.

## 4. 기능 요구사항

### 4.1 구독 관리 (CRUD)
- 추가 / 수정 / 삭제
- 입력 항목: 서비스명, 금액, 통화, 결제 주기, 다음 결제일, 카테고리(드롭다운), 결제수단, 메모
- 다음 결제일은 결제 주기에 따라 자동 계산/갱신
- **프리셋 우선 추가**: 추가 진입 시 `service_presets`(3.2)에서 인기 서비스를 먼저 골라 이름·금액·통화·카테고리·주기를 자동 채운다. 목록에 없으면 “직접 입력”으로 빈 폼. 상세 흐름은 `prd-landing.md 4.3`.

### 4.2 대시보드
- 등록된 구독 목록
- 월 환산 총액 / 연 환산 총액 (통화 혼합 시 결제일 기준 환산)
- 무료 플랜은 등록 개수 표시 (10개 제한 안내 포함)

### 4.3 카테고리별 분석 차트 *(프리미엄)*
- 카테고리별 지출 비중 (도넛/막대)
- 로직(집계)은 공유 패키지, 렌더링만 플랫폼별

### 4.4 갱신 임박 목록
- 다가오는 결제일 순 정렬
- 기간 필터(예: 7일 이내)

### 4.5 푸시 알림
- 결제일/갱신 임박 알림
- 모바일: `expo-notifications` 사용
- 웹: Web Push(서비스워커) — iOS 사파리 제약 있으므로 **모바일 앱 우선** 기능으로 취급

### 4.6 플랜 게이팅 (무료 vs 프리미엄)

| 기능 | 무료 | 프리미엄 |
|------|------|----------|
| 구독 등록 | 최대 10개 | 무제한 |
| 결제일 / 총액 보기 | O | O |
| 갱신 알림 | — | O |
| 카테고리 분석 | — | O |
| 연간 리포트 | — | O |
| 해지 추천 | — | O |

## 5. 화면 정의 (웹 vs 모바일 레이아웃)

내비게이션과 밀도를 플랫폼에 맞게 분리한다. 데이터/로직은 동일, **배치만 다르다.**

| 화면 | 웹 (넓은 화면) | 모바일 (세로) |
|------|----------------|----------------|
| 내비게이션 | 좌측 사이드바 | 하단 탭바 |
| 대시보드 | 총액 카드 + 목록 테이블(다열) | 총액 상단 요약 + 카드 세로 스택 |
| 구독 추가/수정 | 모달 또는 우측 패널 | 전체 화면 폼 |
| 분석 차트 | 넓은 차트 + 범례 옆 배치 | 차트 위, 범례 아래로 스택 |
| 갱신 임박 | 목록 + 필터 사이드 | 목록 + 상단 필터 칩 |

## 6. 요금제 & 결제

- 프리미엄: **월 2,900원 / 연 29,000원** (원문 `2,9000`은 오타로 보아 2,900원으로 확정 — 확인 필요)
- **무료 체험 기간 없음**
- 연간 결제는 약 2개월분 할인 효과

### 결제 수단 — 중요 결정 포인트
- **앱 내에서 디지털 구독을 판매할 경우**, 스토어(App Store·Google Play)는 자체 인앱결제(IAP) 사용을 요구하고 수수료를 가져간다. 웹에서는 외부 결제(예: PG/Stripe 등)를 쓸 수 있다.
- 즉 **모바일 결제 경로와 웹 결제 경로가 달라질 수 있음** → 정책·수수료는 현재 기준으로 재확인 필요 (오픈 이슈).

## 7. 관리자 페이지 (웹 전용)

일반 사용자에게 노출하지 않는 운영자 화면.
- 회원 목록: 각 회원의 플랜, 구독 상태
- 매출 추이
- 해지율(이탈률)
- 프리미엄 이용률
- 접근 권한: 관리자 역할(role)로 제한

## 8. 디자인 원칙

- **정말 심플하고 미니멀.**
- **강조색(accent color) 사용하지 않음** → 무채색(블랙/화이트/그레이) 기반.
- 여백·타이포그래피 중심의 위계.
- 웹/모바일 공통 **디자인 토큰**(색/간격/타이포)을 정의해 일관성 유지.
- 다크모드는 선택 사항.

## 11. 개발 로드맵 (제안)

- **Phase 1 — MVP**: 인증(소셜+이메일), 구독 CRUD, 대시보드, 무료 10개 제한, 모노레포·공유 패키지 세팅
- **Phase 2 — 관리 강화**: 카테고리 분석 차트, 갱신 임박 목록, 푸시 알림(모바일)
- **Phase 3 — 수익화**: 프리미엄 결제, 연간 리포트, 해지 추천
- **Phase 4 — 운영**: 관리자 페이지(회원·매출·해지율·프리미엄 이용률)
